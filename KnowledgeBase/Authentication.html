Authentication


If your organisation does not have a federated identity management solution, you can choose to store user passwords in Workvivo itself. With this approach, users will need to login to Workvivo with their email address and a password of their choosing. The primary benefit of this approach is that it’s the simplest – no configuration is required to get it working. Setting up passwords When an administrator (or automated provisioning process) creates a user account, there is an option to send the user an email with a special link they can use to set up their password.

We also have the option to send out a batch of emails with password setup links to users who do not yet have a password set. This facilitates loading users in bulk from a CSV or Excel file, or through an automated provisioning process. Forgotten password recovery If a user forgets their password, it cannot be recovered as we store passwords using a one-way hashing algorithm which cannot be reversed. Instead, users are sent an email with a one-time link that lets them change their password.

Authentication, Authorisation & Application Security Remember me When logging in to the Workvivo Web application, users have the option to “remember me”. When this option is checked, users will stay logged in indefinitely, until they logout or clear their browser cookies. If this option remains unchecked, users will remain logged in for the duration of the browser session before being logged out. Note that some browsers treat the concept of a browser session differently to others – in some browsers as soon as you close the window it closes the session, with others a session may last until you restart the browser or the entire computer.

Note that the provision of a “remember me” feature can be configured to be switched off at an organisation-level on request. Changing password Users can easily change their password in Workvivo through the “Edit My Profile” section of the application – on both the Web application and on the mobile apps for iOS and Android. Users are asked to enter their old password again in order to reauthenticate before their password is updated. Password hashing Passwords are stored in the Workvivo database as a one-way irreversible hashed value. This means that if someone were to access the user record directly in the database, they would not be able to discover the user’s actual password. The hashing algorithm used in Workvivo is bcrypt, with 10 iterations and a securely generated salt. Delegated Access Workvivo discourages the sharing of passwords – even with trusted colleagues. To provide for a scenario where one user may wish another user to login to Workvivo on their behalf, we provide a delegated access feature. The user essentially selects which users they want to be able to access their account, and that user will be able to switch to the other account from within Workvivo without knowing that user’s password.

Workvivo Staff and User Passwords At Workvivo, we have an internal policy that no member of our staff will ever ask a user for their password. We recommend advising your users to never give anyone their Workvivo password in person, over the phone, by email or through any other means.



User Session Storage When a user logs in to the Workvivo Web application, a unique random session identifier is created for that session. We then store a secure cookie in the user’s browser which is passed to our servers with every request – this allows us to identify the user as they navigate through the application. When a user logs out, this session identifier is destroyed, as is the cookie.

When a user account is deactivated or deleted, their sessions are also deleted which means they will be logged out of the Workvivo application and no longer be able to login until their account is restored. We also have the ability to log a specific user out remotely (without deactivating or deleting their account) if required.




We store user session data in an in-memory Redis cluster, hosted on AWS via the ElastiCache managed service. This cluster is isolated within our own Virtual Private Cloud (VPC) in AWS, and access to it is restricted solely to our Web Application Servers. Data in this store is encrypted both at-rest and in-transit, as are the nightly backups that are automatically created. We retain these backups for one day after which they are automatically destroyed.

API Authentication Our native mobile apps for iOS and Android communicate with the Workvivo back-end via a RESTful JSON API. This is the same API that is used by our Web Application. With the Web app we can leverage cookies to store user state in the browser, but we do not have the concept of cookies in native mobile apps. Instead, we use an OAuth2 compatible API authentication server which generates access tokens that can be used to identify a user on these types of devices. Once an access token has been created, we store this securely on the device and pass this value in with every subsequent API call. The API validates the access token on every request, and if the token is invalid, the API will return an error response.

On the native mobile apps, users remain logged in until they log out or their access tokens are revoked. When a user is deactivated or deleted, their access tokens are automatically deleted preventing them from logging in and automatically logging them out on any devices they have signed in. We can also revoke access tokens for any given user at any time which will effectively log them out of all devices they have signed in on. Logging in on Mobile (SSO environment) For customers using SSO authentication, logging in on mobile has the following process:

1. User enters their email address
2. A Web view pops up in the app that redirects to the identity provider’s sign in page – the user must enter their credentials for the IDM here to login
3. Once the identity provider has authenticated the user they redirect back to Workvivo and the user is now logged in.


Logging in on Mobile (Password environment) The process is slightly different for customers using Password authentication:

1. User enters their email address
2. User enters their Workvivo password
3. Workvivo verifies the user credentials and the user is now logged in.
